## 24 小時打造 MVP

遊戲開始了。

你翻到第二部，準備好了。問題不再是「要不要做」，而是「第一步是什麼」。

答案：**24 小時，讓你的第一個 MVP 上線。**

這不是比喻，不是目標，是一個具體的工作架構。要理解為什麼 24 小時是可能的，先從 Agent 的工作方式說起。

---

### 人類工作是序列的，Agent 工作是並行的

這是一個根本的差異，不是程度上的差異。

人類做事是序列的：做完 A 才開始 B，做完 B 才開始 C。你的注意力在任何時刻只能放在一件事上。就算效率極高，你的工作還是一件接一件完成的。

Agent 不受這個限制。你同時啟動三個 Agent，三個任務同時跑。沒有序列，沒有等待，沒有「做完這個才能做那個」。

這不只是「更快」，是本質上不同的工作模式。用人類的工作邏輯規劃 Agent，是一個根本的錯誤——就像用馬車的邏輯設計汽車，把引擎做成馬的形狀。

24 小時打造 MVP，是並行工作模式的第一個實戰應用。

---

### 傳統 MVP 為什麼要花幾週

傳統創業的 MVP 流程，是人類序列工作的疊加：

```
市場研究（1-2 週）→ 產品設計（2-4 週）→ 技術開發（4-8 週）
→ 網站建設（1-2 週）→ 上線準備（1 週）
```

每個環節等前一個。一個人頻寬有限，無法同時做兩件事；小團隊的協調成本，讓整體更慢，不是更快。

這個序列邏輯，是傳統 MVP 動輒幾個月的根本原因。

---

### AI MVP 的並行架構：三條線同時跑

AILS 的做法不是把序列流程壓縮加速——是**把序列結構整個拆掉**，改成並行：

```
創業家 + Claude Code → Build MVP 產品
Agent 1              → 市場研究（同時進行）
Agent 2              → Agent 原生網站（同時進行）
```

三條線不互相等待。你在建產品的時候，Agent 1 已經在做競品分析，Agent 2 已經在建銷售頁和付款流程。24 小時結束，三條線同時交付：你有產品、有市場情報、有可以銷售的網站。

**第一條線：你 + Claude Code，Build 產品。**

你負責：定義要驗證的核心假設（只有你能做）、決定 MVP 的最小功能邊界、審核每一步的輸出。Claude Code 負責：建技術架構、寫頁面和功能邏輯、整合付款、遇到問題自己除錯。

你們的工作方式是對話式迭代：你給方向，它執行，你審核，它再執行。你不是在寫程式，是在做決策。

**第二條線：Agent 1，市場研究。**

給它明確的任務指令：

```
我正在打造一個 [產品類別]，目標受眾是 [描述]。
1. 找出 3-5 個主要競爭對手，整理定價、核心功能、賣點
2. 搜尋目標受眾在社群裡討論的痛點關鍵詞
3. 根據 OPC 利潤公式（1/2 定價）建議我的 MVP 定價
4. 找出市場上還沒被好好解決的需求缺口
以結構化報告輸出，讓我 24 小時後能直接做定價和定位決策。
```

等你的 MVP 建好，市場研究報告已經等著你了。你立刻根據情報調整定價和定位，而不是建完才開始想「這要賣多少錢」。

**第三條線：Agent 2，Agent 原生網站。**

它在 24 小時內完成：銷售頁（痛點、解法、定價、CTA）、付款流程（整合 Stripe）、SEO 基礎（從上線第一天累積信號）、數據追蹤（每個訪客的行為都被記錄，成為之後 BML 的原始數據）。

### 24 小時的真實樣貌

```
Hour 0：你定義核心假設，同時啟動三條線

Hour 0-24：
  → 你 + Claude Code：建造 MVP 產品
  → Agent 1：市場情報、競品分析、定價建議
  → Agent 2：銷售頁、付款、SEO、數據追蹤

Hour 24：
  → MVP：第一個可運作版本完成
  → 市場報告：競品分析 + 定價建議到位
  → 網站：銷售頁上線，可以開始收訂單
```

第 24 小時結束，你的 MVP 不只是「建好了」——它已經在一個可以銷售、可以追蹤、可以迭代的環境裡跑起來了。

從假設到可以收第一筆錢，24 小時。

有人會問：三條線並行，為什麼不是 6 小時？

因為 24 小時不只是技術時間，它包含一個只有你能做的核心工作：**在建造過程中，根據 Agent 1 的即時市場情報，持續調整方向**。某個對手的定價比你想的低、某個痛點比預期強、某個功能根本沒人要——你要根據這些情報，和 Claude Code 動態調整 MVP。這個「判斷 + 調整」的過程需要時間。不是技術的時間，是思考和決策的時間。

24 小時，是讓這個過程有足夠空間、同時保持緊迫感的正確窗口。

### 三條線是起點，不是上限

三條線是你的**啟動配置**。當 MVP 上線、第一批市場信號進來，你不需要等、不需要招人——直接新增 Agent 工作流。

**新增銷售 Agent**：有人看了定價頁但沒購買？銷售 Agent 24 小時監測訪客行為，識別高意向訪客，自動觸發三天的信任建立 Email 序列，即時回覆銷售頁問題，追蹤每個潛在客戶的轉換進度。這是以前需要一個業務人員做的事——全天候，不用你盯。

**新增社群運營 Agent**：MVP 上線後有了真實數據，這是最好的內容素材。社群 Agent 每週從網站數據擷取值得分享的洞察，整理成 Building in Public 內容——本週流量、哪個功能最多人感興趣、你做了什麼迭代——自動發布到社群。你每週只花 30 分鐘回應最有價值的互動。

業務推進一步，AI 工廠就擴充一步。每一層新需求，當天就有對應的 Agent 接手。

### 速度，是 Agent 對比人才的絕對優勢

傳統創業的擴張，被一個隱形成本綁架：招募和培訓。

```
寫職缺（1 週）→ 等履歷（2-4 週）→ 面試（2-4 週)
→ 發 Offer、等到職（2-4 週）→ 入職培訓（1-3 個月）
```

從「我需要一個業務」到「這個業務開始產生業績」，快則三個月，慢則半年。招錯人，重來一輪。整個過程，市場機會不會等你。

Agent 的擴張邏輯：

```
識別需求（1 小時）→ 定義工作流（幾小時）→ 啟動（當天）→ 開始工作（即刻）
```

今天發現銷售漏斗有摩擦，今天新增銷售 Agent，今晚它已經在追蹤訪客、發跟進 Email。它從啟動那一刻就讀取了你的產品、受眾、品牌語氣——不需要入職流程。而且它不請假、不離職、不要求加薪、不搞辦公室政治。

這帶來的不只是速度差距，是策略靈活度的根本差異。

招募人才，是一個以月計的長期承諾——所以傳統公司傾向保守，不敢輕易試新方向，方向錯了就是沉沒成本。新增 Agent，是一個今天就能驗證的實驗——方向不對，調整指令或停掉工作流，沒有資遣費，沒有人際摩擦。

**招人，你在做承諾。加 Agent，你在做實驗。**

## 零成本的 MVP

### 你以為要花錢的，其實不用

傳統建一個產品網站的成本：工程師月薪 6-10 萬，開發 1-3 個月。伺服器每月一千到一萬。資料庫再加幾百到幾千。域名、SSL、CDN……

在產品有第一個付費用戶之前，你已經燒掉幾十萬。

這個現實在 2026 年已經過期了。現在，你可以用開源工具搭配商業服務的免費方案，把一個完整可運作的 MVP 建起來——從開發到上線，成本趨近於零。

不是功能殘缺的玩具。是可以真實獲客、展示產品、收集名單、完成付款的完整系統。

---

### 為什麼能做到零成本

兩個關鍵的結構。

**開源生態系已經成熟。** 前端框架、後端框架、資料庫工具、UI 組件——都有品質極高的開源版本，免費，有龐大社群維護。

**商業服務的免費方案已經夠用。** 主流雲端服務的設計原則是：你還小的時候免費用，業務長大超過門檻再收費。對 OPC 來說這非常友善——MVP 階段的流量和用戶數，完全在免費範圍內。等你需要付費，表示業務已經驗證成立、有了收入。

---

### 零成本 MVP 工具組合

**前端**：Next.js（最主流的 React 框架，Claude Code 對它理解最深）＋ Tailwind CSS（Agent 非常擅長，語法接近自然語言）＋ shadcn/ui（高品質組件庫，直接用）。全部開源免費。

**部署**：Vercel——`git push` 之後自動部署，30 秒上線，自動 HTTPS 和 CDN。免費方案 100GB 頻寬/月，MVP 完全夠用。替代選項 Cloudflare Pages，免費方案無限頻寬。

**資料庫**：Supabase——開源的 Firebase 替代品，PostgreSQL + 驗證 + 儲存，免費方案 500MB 資料庫、50,000 月活用戶。替代選項 Neon，閒置自動睡眠，真的不用為閒置付費。

**會員登入**：Clerk——幾行程式碼完成登入頁和社交登入，免費方案 10,000 月活。想完全自主控制就用 NextAuth.js，開源無上限。

**表單與 Email**：Formspree（表單提交直接進信箱，免費每月 50 次）＋ Resend（Email 發送，免費每月 3,000 封）。

**付款**：Stripe——沒有月費，只收成功交易的手續費。開始有收入之前，它真的是零成本。完整的測試模式讓你上線前就能驗證整個付款流程。

**儲存**：Cloudflare R2（免費 10GB，沒有出流量費）或 Supabase Storage（附在免費方案內）。

### 四種情境的完整技術棧

**情境一：展示頁 + 等待名單**（今天就能上線）
Next.js + Tailwind → Vercel → Formspree。總成本：$0。

**情境二：銷售頁 + 付款**
Next.js + Tailwind + shadcn/ui → Vercel → Stripe。總成本：$0，成交才有手續費。

**情境三：有會員功能的課程平台**
Next.js + Supabase + Clerk + Resend → Vercel → Stripe。總成本：$0。

**情境四：有 AI 功能的 Agent 原生產品**
上述組合 + Claude API（按使用量計 Token 費）。總成本：只有 Token 費。

免費方案不是永遠夠用，但升級是好信號，不是壞消息。

## 先銷售，再建造

### 精實創業最反直覺的一句話

大多數人的創業邏輯：先做出夠好的產品 → 再找客戶 → 然後賺錢。

看起來合理。要賣一個東西，應該先有那個東西。

但這個邏輯有一個根本問題：**你不知道你做出來的東西，有沒有人要買。**

你花三個月做課程，可能沒人報名。花六個月開發 SaaS，可能沒人付月費。這種情況的頻率高得驚人——42% 的新創失敗原因是「市場沒有需求」。不是做不出產品，是做出來沒人要。

先建造再銷售，是把大量時間賭在一個未驗證的假設上。

**先銷售再建造，是把風險降到接近零。**

---

### 讓市場決定你該建什麼

先銷售的核心，是把驗證放在建造之前。你先讓市場告訴你「我願意為這件事付錢」，然後才開始建造。

這個順序改變，帶來三個結果：

**你只建造有人要買的東西。** 沒有人預購，你不建。省下的製作時間，直接去測下一個假設。

**你建的東西從一開始就符合市場期待。** 第一批付費客戶就是第一批真實用戶。交付之前你可以直接問他們：「你付錢是期待解決什麼問題？」你的產品從第一天就有精準的方向。

**你有現金流支撐製作成本。** 先收錢，再花錢建造。不需要融資，不需要從口袋掏錢。

---

### 先銷售不等於詐騙

一個常見的誤解要說清楚：先銷售不是賣一個不存在的東西騙人。

正確的方式是誠實預購：

「這個產品正在建設中，計劃 X 月交付。預購享 Y 折優惠。如果交付時不符合期待，全額退款。」

三個關鍵元素：**誠實**（明確說明還沒建好）、**承諾**（說清楚交付時間和內容）、**保障**（退款保證，讓買家風險趨近於零）。

書籍、課程、軟體都有預購先例。你不是在做激進的事，你在用精實創業最核心的邏輯：先確認需求，再投入製作。

---

### AI 給你的新能力：以假亂真

傳統「先銷售」有一個隱藏門檻：你要讓客戶相信產品是真實的、能交付的。這需要展示材料——清晰的產品頁、有說服力的文案、視覺化的呈現。沒有這些，很難說服人預購一個看不到的東西。

AI 把這個門檻砍掉了。

Claude 兩小時寫出完整的產品銷售頁。Claude Code 半天把它架成一個看起來完整的網站。AI 生成的產品展示圖，讓頁面視覺上就像一個已經存在的產品。

**這是 AI 時代的「以假亂真」能力**——不是欺騙，是讓你的驗證材料在產品存在之前，就達到讓市場做判斷的品質門檻。

對一人電商公司，這個能力的意義尤其重大。

以前驗證一個電商產品，幾乎必須先製造實體庫存：備貨、攝影、建 SKU、設物流——幾十萬起跳，賭一個你不確定市場要不要的東西。

現在：用 AI 建一個完整的商品頁——描述、規格、AI 生成的設計圖、定價、購買按鈕。設定驗證門檻：「100 個人訂購就正式生產。」

有 100 人下單，你拿訂單去工廠談最小生產量，用預購款覆蓋製造成本。沒有 100 人，退款，零庫存，沒有損失。

**虛擬上市驗證市場，有訂單才生產**——這是 AI 給一人電商的作弊碼。

### OPC 版本的先銷售：極端但合理

OPC 的先銷售可以做到多極端？

你今天有一個想法。Agent 下午建好預購頁。晚上你發給 10 個認識的目標受眾。

明天有 3 個人預購：假設初步驗證，開始建造。沒有人預購：你花了半天和幾百元 Token 費，知道了這個方向沒市場，明天換下一個假設。

傳統創業者花六個月才能得到這個答案，還要承受六個月的損失。OPC 在 24 小時內知道，幾乎沒有代價。

不是 OPC 創業家更聰明。是工具和成本結構，讓先銷售變得沒有摩擦、可以重複、快速迭代。

---

### 三個常見場景

**你有知識，想做課程。** 不要先錄課。先做預購頁：課程名稱、核心內容、對誰有幫助、交付時間、預購價。發給社群說「有足夠的人預購就正式製作」。有人預購，你開始錄——你知道有人在等，知道他們的期待。沒人預購，你省下整段錄課時間，去問沒買的人為什麼不買。

**你有技術，想做工具。** 不要先開發完。先做展示頁：截圖、這個工具解決什麼問題、定價、等待名單。0 個人加入等待名單，比開發六個月後才發現沒人要，節省太多了。

**你有服務能力，想做顧問。** 最直接的先銷售：發一份提案，直接問對方要不要雇用你。不需要網站，不需要品牌。一份寫清楚「我能幫你解決什麼、怎麼解決、費用多少」的文件就夠了。Agent 幫你寫得精準專業，你判斷要發給誰、收多少錢。

---

### 先銷售之後：你欠客戶的事

先銷售有一個配套責任。

你收了預購款，你欠客戶一個你承諾過的產品。

這不是壓力，是動力。有真實的客戶在等，是世界上最強的交付動力。你不是在做一個「也許有一天會完成」的東西，你在給一個付了錢的真實的人交付承諾。

而且，你可以讓第一批客戶成為共同創造者：問他們問題，讓他們看到進度，在建造過程中收他們的回饋。他們的投入讓產品更好，他們的早期參與讓滿意度更高，他們的滿意變成你第一批真實的口碑。

先銷售，讓整個過程從「一個人在做東西」，變成「一群人在一起打造一件值得做的事」。

---

### 今天的第一步

用兩句話寫出你的預購訊息：

「我在做 [一句話描述]，現在開放 [N] 個早鳥名額，定價 [X]。」

不用發出去，先寫出來。寫完問自己：把這兩句話傳給五個認識的人，他們有沒有可能問「這是什麼？我想了解」？

能寫出讓人想繼續問的兩句話，就是你假設夠具體的第一個測試。

---

## 創造顧客，銷售成功

### 銷售不是等待，是主動創造

很多人把銷售想成一個被動的過程：做好產品，放上網，等人找上來。

這個邏輯在流量充足的時代可能成立。但對一個剛上線 MVP 的 OPC 創業家來說，等待是最昂貴的策略。

你的 MVP 上線了，沒有人知道它存在。SEO 要幾個月才發揮作用，廣告要預算，口碑要第一批客戶。在這些機制建立之前，你唯一能做的是：**主動創造顧客**。

不是等人來找你。是你去找到可能需要你的人，讓他們理解你的產品能幫他們做什麼，讓他們做出購買決定。

---

### 創造顧客的三個層次

每一個付費顧客在付款之前，都經歷了從「不認識你」到「決定買」的過程。三個層次：

**層次一：讓對的人知道你存在（Awareness）。**

目標受眾不知道你的產品存在，定價再低、品質再好，都不會有銷售。Agent 能做的：自動產出 SEO 內容，讓受眾搜尋問題時找到你；在社群持續發布有價值的內容；把產品資訊發布到對的社群和平台。

**層次二：建立信任，讓人考慮購買（Consideration）。**

知道你存在，不等於相信你能幫到他。OPC 最有效率的信任建立方式是 BIP（Building in Public）——把真實過程、真實數據、真實失敗和修正公開分享。一個看著你從第一天做到現在的人，他的信任是親眼見證出來的，不是被廣告說服出來的。

Agent 把你的 BML 數據和迭代過程自動整理成可發布的內容，持續更新。信任的積累，自動發生。

**層次三：讓猶豫的人做出決定（Decision）。**

有人看了銷售頁，有興趣，但還沒買。猶豫的原因通常是：不確定適不適合自己、不確定是不是好時機、沒有足夠強的理由現在行動。

Agent 能做的：即時客服 Agent 24 小時回答訪客問題；Email 序列在接下來幾天送出解除疑慮的內容；銷售頁展示真實的早期客戶案例，讓猶豫的人看到「跟我類似的人也做到了」。

---

### 「銷售成功」的 OPC 定義

傳統銷售的成功定義很簡單：賣出去。

OPC 的標準更高：**客戶付了錢，而且得到了他付錢時期待的結果。**

這個定義決定你的商業模式上限。

如果成功只是賣出去，你的選項只有兩條：不斷找新客戶，或靠訂閱留住舊客戶——兩條都需要持續的行銷投入。

如果成功是客戶達到了預期結果，你多了第三條路：**讓每個成功的客戶，帶來下一個客戶。**

一個因為你的服務真的賺到錢的客戶，會主動告訴他的朋友。這個推薦不花你一毛行銷費。它發生的前提是：你的銷售成功，不只是你收到錢，而是客戶確實得到了他買的東西。

創造顧客讓你有收入。銷售成功讓你有口碑和可持續的生意。

---

### 從賣產品到賣成功

CH01-3 說了利潤公式的第一階段：1/2 定價、1/10 成本、95% 毛利。這是固定定價的產品銷售：賣一個產品，客戶付錢，你交付，完成。

第二階段，是當你能穩定幫客戶達到成果之後才能進入的模式：**按成果收費**。

用本書的核心案例：你幫客戶建一套 Agent 原生獲客變現系統。第一階段的做法是收 5 萬建設費，建完結束。第二階段的做法是：收少量或不收建設費，但約定「當你的網站月收入因為這套系統超過 10 萬，我抽 15%」。

三個結構優勢：

**目標完全對齊。** 你越想幫客戶賺更多，你自己賺更多。真正的利益共同體，不是買賣關係。

**收入天花板打開了。** 固定收費有市場認知的上限。按成果分潤，客戶的成果持續擴大，你的收入持續擴大，不需要不斷找新客戶。

**客戶數更少，關係更深。** 10 個深度綁定的成功客戶，比 100 個一次性交易的客戶帶來更高、更穩定的收入。而且每個成功客戶都是你最強的銷售素材。

---

### Agent 工廠怎麼支撐這個模式

「主動找客戶、建立信任、持續服務每個客戶到成功」聽起來需要大量人力。傳統顧問公司正是因為這樣，規模很難擴大。

OPC 用 Agent 工廠解開這個矛盾：

**Agent 全天候創造顧客。** 網站每天發布內容、24 小時回應訪客、自動培養名單。你睡覺的時候，工廠在工作。

**Agent 監測每個客戶的成功進度。** 為每個客戶部署成果追蹤儀表板，Agent 自動讀取流量、轉換率、收入變化，有異常就通知你。你不需要親自盯每個客戶。

**Agent 把成功複製。** 一個客戶達到好結果，Agent 分析他的成功路徑，整理成可複製的模板。下一個客戶從第一天就站在已驗證的方法上。

你的角色不是服務每一個需求。是設計讓客戶成功的系統，讓 Agent 工廠去執行和監測。

---

### 今天的第一步

理論說夠了。現在做一件事。

列出你認識的人裡，符合目標受眾描述的 3 到 5 個人。不需要是最親近的朋友，需要是確實有你想解決的那個問題的人。

傳給他們：

「我在做一個 [一句話說清楚]，目標是幫 [受眾] 解決 [具體問題]。你有這個問題嗎？有的話，我想和你聊 15 分鐘，了解你的情況，看看我的方案是不是真的能幫到你。」

不是在賣，是在了解。這 15 分鐘的對話，讓你聽到他們用什麼語言描述問題、真正的痛點在哪、對解法的期待是什麼。比你自己想像出來的受眾分析，精準 10 倍。

**創造第一個顧客，從一個對話開始。**

## AI MVP 實戰：用 Claude Code 打造一項產品

### 停止看教程，開始實際做

前面四章說了 AI MVP 的邏輯和框架。這一章不說框架。

我們打開 Claude Code，實際做一個產品，從零到上線。這個流程你可以直接套用在自己的產品上。

---

### 我們要做什麼

示範產品，就是本書從第一章討論到現在的那個：

**「Agent 原生獲客變現網站」建置服務。**

真實的市場需求：很多中小企業主和創作者，想要一個讓 Agent 24 小時自動運營的網站，但不知道怎麼做，也沒有工程師資源。

- 目標客戶：有明確產品的個人品牌 / 小型電商 / 知識型創業者
- 競爭對手市價：傳統網頁設計公司，10 萬起跳
- OPC 定價：5 萬（對手的 1/2）
- 交付成本：Claude Code + Vercel + Token 費，幾千元以內（對手的 1/10）

這個產品本身就是一個 Agent 原生系統——我們用 Claude Code 建造它，用同一套技術交付給客戶，自己的網站就是最好的 Demo。

---

### 步驟一：定義你要建什麼

打開 Claude Code 之前，先把目標說清楚。這是最重要的一步，也是最多人跳過的一步。

寫下這份文件：

```
我要建一個 MVP 版本的「Agent 原生獲客變現網站」展示頁，
用來驗證市場是否願意為這個服務付費。

頁面需要：
1. 清楚說明這個服務解決什麼問題
2. 展示三個核心功能（Agent 內容自動化、Agent 客服、即時數據儀表板）
3. 明確的定價區塊（5 萬元，含說明）
4. 一個聯絡 / 預購表單

技術要求：
- Next.js + Tailwind CSS
- 部署到 Vercel（免費方案）
- 不需要後端，表單用 Formspree
- 行動裝置正常顯示
```

模糊的指令，模糊的結果。清楚的指令，可預期的結果。

---

### 步驟二：讓 Claude Code 建立架構

打開終端機，進入工作目錄，啟動：

```bash
claude
```

把需求文件貼給它。Claude Code 會建立 Next.js 專案、設定 Tailwind、建頁面組件、寫出基礎內容。過程中它自己執行命令、建立檔案、遇到問題自己除錯。

你的工作是觀察，在它問問題的時候給出判斷。不需要自己打任何程式碼。

---

### 步驟三：審核輸出，給出品味判斷

基礎版本完成後，在 localhost:3000 預覽。看三件事：

**訊息是否清楚？** 一個完全不認識你的人，30 秒內能不能理解這個服務能幫他解決什麼問題？

**定價是否清楚可見？** 很多頁面把定價藏在最底部。定價應該在頁面上半部就出現。

**CTA 是否明確？**「立即預購」比「了解更多」好。「預購現省 30%，本週限定」比「立即預購」更好。讓人知道現在應該做什麼，而且應該現在做。

把觀察告訴 Claude Code：「標題不夠直接，改成更聚焦客戶痛點的版本」「定價區塊放大，加一個比較表格」「CTA 換更緊迫的文案」。

這個來回，就是你作為 AI 工廠老闆在做的事：給方向、審核結果、做品味判斷。

---

### 步驟四：部署到 Vercel

對頁面滿意了，告訴它：

```
請幫我把這個專案部署到 Vercel，給我一個公開的 URL。
```

Claude Code 引導你完成部署——建 git 倉庫、推送、連結 Vercel。完成後你得到一個公開 URL。

這個 URL，就是你的 MVP 上線了。從開始到這一步，幾個小時。

---

### 步驟五：第一波推廣，收集真實信號

URL 有了，開始推廣。不是買廣告，不是做 SEO。是直接找人。

列出你認識的人裡最有可能是目標客戶的 5 到 10 個，傳給他們：

「我在做一個新服務，幫有產品的創作者建一套讓 AI 24 小時自動獲客和銷售的系統。這是展示頁：[URL]。你覺得這解決了你一個真實的問題嗎？對定價有什麼想法？」

注意：不是要他們幫你傳播，是直接問真實反應。

48 小時後看結果：

- 「我有興趣，怎麼預購？」→ 假設初步驗證，回到 CH04-3 的預購流程
- 「我看不懂這個服務在做什麼」→ 訊息不夠清楚，回去改頁面
- 「定價太高」→ 調整定價，或者你找的人不是對的受眾
- 沒有人回應 → 訊息沒有擊中痛點，或受眾找錯了

每一種反應都是有用的信號。沒有「失敗」，只有「學到了什麼」。

---

### 你剛剛做了什麼

回頭看這個流程：

你沒有學一年的程式。沒有花幾十萬請設計公司。沒有等三個月才上線。你在幾個小時內，建出了一個可以對外展示、可以收集市場信號的 MVP。

它不是最終版本，也不需要是。它的品質，足以讓一個潛在客戶理解你的服務、判斷它是否解決他的問題、決定要不要進一步了解。

有了真實信號，你的下一輪迭代才有根據。你不是在猜客戶想要什麼，你是在根據真實反應做決定。

**用 Claude Code 快速建造，用 Agent 原生網站即時測試，用真實行為數據驅動下一步。從想法到上線，24 小時。**

同樣的流程適用於：課程銷售頁、SaaS 工具 MVP、電子報訂閱頁、數位產品商店、個人品牌網站。技術不是門檻，清楚的目標才是關鍵。

---

### 今天的第一步

打開 Claude Code，輸入：

「幫我建一個最小的 [你的產品類型] 銷售頁，包含：標題、三個核心賣點、定價、一個 CTA 按鈕。用 HTML 和 Tailwind CSS。」

不要只看它輸出什麼。注意它問了什麼問題、做了什麼假設、在哪裡需要你確認方向。

這就是你和 AI CEO 第一次真正協作的樣子。
